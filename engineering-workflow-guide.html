Engineering Workflow Skills Guide

A practical map for using the installed Engineering Workflow skill set. For a new repo, run setup once. For project work, start at step 1 with grill-with-docs, then specify, ticket, and deliver one ticket at a time.

To start a new project in an already prepared repo, begin at step 1: grill-with-docs. To start in a repo that has never used this workflow, run setup-engineering-workflow once first, then move into step 1. After to-tickets, use deliver-ticket for one frontier ticket at a time: deliver it to PR, finalize that same ticket when checks are ready, then return to ticket selection.

Operating Rules

These rules keep the workflow useful instead of ceremonial.

Loop one ticket at a time

After to-tickets, repeat deliver-ticket for one frontier ticket, finalize it, then choose the next unblocked ticket. Do not roll into the next ticket automatically.

Keep tickets independently shippable

Each ticket should have a clear user or system outcome, known blockers, verification steps, and enough context for deliver-ticket to run in a focused Codex thread.

Review against the promise

deliver-ticket uses code-review as the gate: standards and spec alignment both need to be checked before a ticket is finalized.

Skill Details

This section explains each skill in the updated Engineering Workflow set and how it hands off to the next step.

0 setup-engineering-workflow

The repo-level setup step. It prepares the environment where the other skills can publish issues, label work, and store domain documentation consistently.

Use when
Starting the workflow in a new repo or normalizing an existing repo before planning work.
Output
Issue tracker conventions, triage labels, and a predictable domain-doc layout.
Hands to
Any planning or delivery skill, especially grill-with-docs, triage, to-spec, and to-tickets.

A wayfinder

The large-work navigator. It is for work too broad or uncertain for a single conversation, spec, or ticket set.

Use when
The destination is clear enough to care about, but the route requires investigation across multiple sessions.
Output
A map of investigation tickets and a path toward a spec once enough unknowns are resolved.
Hands to
research, prototype, grill-with-docs, or to-spec depending on what each investigation finds.

1 grill-with-docs

The recommended default starting skill for a product idea or design. It sharpens the plan by asking hard questions and records the answers as durable docs.

Use when
You have an idea, plan, or design that needs pressure before it becomes a spec.
Output
Clarified scope, assumptions, risks, glossary updates, and architectural decision records where useful.
Hands to
to-spec when the plan is crisp, or research/prototype/codebase-design if unanswered risks remain.

S1 grilling

The lighter pressure-test interview. It asks direct questions about a plan or design without making documentation the main output.

Use when
You need a quick challenge session, or you want to stress-test a narrow plan before documenting it.
Output
Sharper assumptions, exposed gaps, and clearer acceptance criteria.
Hands to
grill-with-docs if the answers should become durable docs, or to-spec if enough clarity already exists.

S2 domain-modeling

The vocabulary and decision-capture skill. It keeps the project language stable as the product and codebase evolve.

Use when
Terms are ambiguous, domain concepts are drifting, or a decision deserves a durable record.
Output
Glossary entries, canonical terms, clarified relationships, and architecture decision records.
Hands to
grill-with-docs, to-spec, codebase-design, and implementation tickets that need shared language.

S3 research

The evidence-gathering skill. It answers factual questions using high-trust sources and saves the result so planning does not rely on memory.

Use when
External docs, APIs, product behavior, legal constraints, standards, or current facts affect the design.
Output
A Markdown research note with findings, sources, and implications for the work.
Hands to
grill-with-docs, to-spec, prototype, or deliver-ticket depending on whether the finding changes design or execution.

S4 prototype

The throwaway proof skill. It builds a small artifact to answer a specific design, UI, state, or logic question.

Use when
A cheap experiment would reveal more than more discussion, especially for interaction, state, or feasibility questions.
Output
A disposable prototype and a decision about what should or should not move into production work.
Hands to
grill-with-docs or to-spec after the prototype resolves the uncertainty.

S5 codebase-design

The module-design vocabulary skill. It helps decide where interfaces, boundaries, and deeper modules should live.

Use when
You are designing or improving an API, making code more testable, or deciding how a feature should fit the existing architecture.
Output
Clearer module boundaries, simpler interfaces, and implementation guidance for maintainable code.
Hands to
to-spec, to-tickets, deliver-ticket, or improve-codebase-architecture for broader architecture work.

2 to-spec

The synthesis step. It turns the current conversation and decisions into a spec without restarting the discovery process.

Use when
The idea is clear enough that future work needs a stable written contract.
Output
A spec or project issue covering problem, goals, scope, requirements, constraints, and acceptance criteria.
Hands to
to-tickets for implementation slicing, or back to grilling/research if the spec exposes gaps.

3 to-tickets

The decomposition step. It converts a spec or plan into implementation tickets that can be executed one at a time.

Use when
A spec exists and the next need is a sequence of concrete, testable work slices.
Output
Tracer-bullet tickets with blockers, dependencies, expected behavior, and verification guidance.
Hands to
deliver-ticket, one frontier ticket at a time.

4 deliver-ticket

The ticket delivery loop. It wraps the repeated branch, implementation, TDD, verification, review, PR, and finalization steps for exactly one ticket.

Use when
A ticket is ready to build, a next frontier ticket needs to be chosen, or an existing ticket PR needs finalization.
Output
A delivered ticket branch and PR, or a finalized ticket with merge/completion status and remaining blocker if any.
Hands to
implement, tdd, and code-review internally; then back to ticket selection after the current ticket is complete.

B triage

The intake skill. It moves issues and external PRs through a clear state machine until each item is actionable or rejected.

Use when
Incoming work needs categorization, reproduction, clarification, or an agent-ready brief.
Output
Verified issue state, labels, notes, and a clear next action for planning or implementation.
Hands to
grilling if unclear, diagnosing-bugs if broken behavior is involved, or deliver-ticket when ready.

C diagnosing-bugs

The debugging loop. It protects implementation from guesswork by first reproducing and explaining the failure.

Use when
Something is broken, slow, flaky, throwing, failing tests, or behaving differently than expected.
Output
A reproducible failure, narrowed cause, fix strategy, and verification plan.
Hands to
deliver-ticket for the fix when the bug has been turned into an agent-ready ticket.

D improve-codebase-architecture

The architecture improvement scout. It looks across a codebase for opportunities to create deeper modules and cleaner boundaries.

Use when
The product works, but the code is getting harder to change, test, or reason about.
Output
A visual report of improvement candidates and a selected path for deeper design work.
Hands to
codebase-design, grill-with-docs, to-spec, or to-tickets depending on the size of the improvement.

I implement

The build component inside the ticket loop. It executes a spec or ticket while respecting the repo's patterns, tests, and verification expectations.

Use when
A ticket, spec slice, or bug-fix plan is ready to build.
Output
Code changes, focused tests, verification results, and a clear account of what changed.
Hands to
deliver-ticket for review, PR, and finalization, or diagnosing-bugs if verification reveals a deeper failure.

T tdd

The test-first discipline. It gives implementation a red-green-refactor loop when behavior can be specified before code is changed.

Use when
Building features or fixing bugs where tests can describe the expected behavior or regression.
Output
A failing test first, a minimal implementation to pass it, and a refactor after behavior is protected.
Hands to
deliver-ticket during ticket delivery, then code-review for standards and spec alignment.

R code-review

The quality gate inside the ticket loop. It reviews the diff against both the repo's engineering standards and the promise made in the spec or ticket.

Use when
A branch, PR, or work-in-progress diff needs review before merge or before continuing.
Output
Findings ordered by severity, grounded in file and line references, plus any residual risk or test gaps.
Hands to
deliver-ticket for fixes, PR publication, or final merge work after the review is clean.

H codex-handoff

The context-transfer skill. It prepares a fresh Codex thread to continue the work without dragging the whole conversation along.

Use when
The current thread is long, the work should continue elsewhere, or a forked Codex thread should pick up a specific task.
Output
A concise, redacted summary of goal, context, decisions, completed work, verification, remaining work, and next action.
Hands to
A forked or new Codex thread that can continue from the summary and invoke the right next skill.
